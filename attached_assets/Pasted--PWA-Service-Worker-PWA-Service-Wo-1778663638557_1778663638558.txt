בקשה לקודקס — עדכון PWA ו־Service Worker

יש לבצע בדיקה ועדכון מלאים של מנגנון ה־PWA וה־Service Worker באתר.

המטרה:
לוודא שהאתר מתנהג כמו אפליקציה תקינה, גם במובייל וגם בדסקטופ, כולל התקנה למסך הבית, טעינה יציבה, אייקונים נכונים, manifest תקין ו־Service Worker מעודכן.

מה לבצע:

1. בדיקת Manifest

יש לבדוק את קובץ ה־manifest ולוודא שהוא תקין ומוגדר נכון.

נדרש לבדוק:

* `name`
* `short_name`
* `description`
* `start_url`
* `scope`
* `display`
* `background_color`
* `theme_color`
* מערך `icons`

השם הרצוי לאתר:
`Lior's Pâtisserie`

אם מדובר באדמין:
`Admin Pâtisserie`

חשוב:
מכיוון שהאתר רץ תחת GitHub Pages בנתיב:

`/Lior-patisserie/`

צריך לוודא שכל הנתיבים ב־manifest מותאמים ל־base path הזה ולא נשברים.

2. בדיקת אייקונים

יש לוודא שכל האייקונים קיימים, נטענים ולא מחזירים 404.

אייקונים לאתר הראשי:

* `favicon.ico`
* `favicon-16x16.png`
* `favicon-32x32.png`
* `apple-touch-icon.png`
* `android-chrome-192x192.png`
* `android-chrome-512x512.png`

אייקונים לאדמין, אם יש manifest נפרד או הגדרה נפרדת:

* `admin-512x512.png`
* `admin-192x192.png`
* `admin-apple-touch-icon.png`
* `admin-favicon-32x32.png`
* `admin-favicon-16x16.png`

3. עדכון Service Worker

יש לעדכן את ה־Service Worker כך שלא ייתקע על גרסה ישנה של האתר.

דרישה:

* להעלות מספר גרסה / cache version.
* למחוק caches ישנים בזמן `activate`.
* לוודא ש־Service Worker חדש נכנס לפעולה אחרי deploy.
* להוסיף `skipWaiting()` ו־`clients.claim()` אם מתאים למבנה הקיים.
* לוודא שאין מצב שבו המשתמש רואה קבצים ישנים אחרי שהאתר עודכן.

דוגמה לכיוון:
להגדיר שם cache עם גרסה חדשה, למשל:

`lior-patisserie-cache-vNEW`

ולמחוק cache ישן שלא תואם לגרסה החדשה.

4. אסטרטגיית Cache

יש לבדוק שה־cache לא גורם לבעיות בתמונות, תוכן דינמי או עדכוני Supabase.

חשוב:

* לא לשמור ב־cache בצורה קשיחה נתונים שמגיעים מ־Supabase אם הם צריכים להתעדכן באדמין.
* לא לגרום לכך שתוכן שנערך באדמין ימשיך להציג גרסה ישנה באתר.
* לקבצי static כמו CSS, JS, fonts ואייקונים אפשר להשתמש ב־cache.
* לתוכן דינמי / API / Supabase עדיף network-first או לא לשמור cache אגרסיבי.

5. התאמה לאדמין

יש לבדוק שגם עמוד האדמין עובד תקין כחלק מחוויית האפליקציה.

נדרש:

* לוודא שהאדמין לא נשבר בגלל Service Worker.
* לוודא שלא נשמרים פרטי התחברות או מידע רגיש ב־cache.
* לוודא שאחרי עדכון קוד, האדמין לא ממשיך להציג גרסה ישנה.
* אם יש צורך, להחריג נתיבי admin מסוימים מה־cache.

6. בדיקות חובה אחרי העדכון

לאחר השינוי יש לבדוק:

בדסקטופ:

* פתיחת האתר.
* רענון קשיח.
* בדיקה ב־DevTools תחת Application.
* בדיקת Manifest.
* בדיקת Service Worker.
* בדיקת Cache Storage.
* לוודא שאין שגיאות 404 לאייקונים / manifest / service worker.

במובייל:

* פתיחה בדפדפן.
* הוספה למסך הבית.
* פתיחה כאפליקציה.
* בדיקה שהאייקון נכון.
* בדיקה שהשם נכון.
* בדיקה שהאתר נפתח בנתיב הנכון ולא נשבר.
* בדיקה שהכפתורים והאדמין עובדים.

7. לאחר deploy

לאחר העלאה ל־main / GitHub Pages:

* לפתוח את האתר בחלון חדש.
* לבצע רענון קשיח.
* אם צריך, להסיר את האפליקציה מהמכשיר ולהתקין מחדש לבדיקה נקייה.
* לוודא שה־Service Worker החדש נרשם.
* לוודא שה־cache version החדש פעיל.

תוצאה רצויה:
האתר מותקן ועובד כמו אפליקציה תקינה.
האייקונים מופיעים נכון.
שם האתר והאדמין נכונים.
ה־Service Worker לא נתקע על גרסה ישנה.
תוכן שמתעדכן באדמין לא נשאר תקוע ב־cache.
אין שגיאות Manifest / PWA / SW בקונסול.
